# 파이프라인 평가 보고서 (2026-06-10)

> 평가 방법: 전체 소스 코드 정적 분석 + 최근 6개 실행(`output/raw`, `output/drafts`)의 실데이터 통계 검증.
> 파이프라인을 실행하지 않고, 이미 저장된 산출물에서 수치를 추출해 코드의 동작과 대조했다.

---

## 총평

수집 → 원문 확보 → 단계적 생성 → 검증 루프의 골격은 잘 설계되어 있고, 16차까지의 토큰 최적화
(중복 배분 축소, 메모 기반 요약, Selective Opus, 라이브러리 캐시)도 실데이터에서 효과가 확인된다.

그러나 **"많이 모은 자료 중 무엇을 보고서에 넣을 것인가"를 결정하는 선별 단계가 사실상 망가져 있다.**
수집은 250~320건씩 하는데 보고서에는 신뢰도 상위 40건만 들어가고, 그 40건이 특허·R&D 과제 목록·통계
스텁 같은 **수백 자짜리 메타데이터로 도배**되어 정작 논문·웹 분석자료(전체의 80%)는 한 건도 안 쓰인다.
더 심각하게는, **검증 루프가 재수집한 자료가 보고서에 0건 반영되는 버그**가 있어 검증 루프의 비용
전체(agy 2회 + 재수집 + 원문 저장 + 챕터 재생성 ≈ 회당 $1.2+)가 헛돈이 되고 있다.

즉 현재 가장 큰 낭비는 "프롬프트에 토큰을 너무 많이 넣는 것"이 아니라
**"비싸게 모은 자료의 85%+를 버리고, 보완 루프는 아무것도 보완하지 못하는 것"**이다.

## 치명적 문제 (즉시 수정 권장)

### C1. 검증 루프가 재수집한 자료가 보고서에 단 한 건도 반영되지 않는다 (버그)

**현상**: 06-10 두 런 모두 재수집 신규 158건·149건이 보고서 입력에 0건 진입.

**원인 (코드 추적)**:
1. [report_validator.py:369](../src/validators/report_validator.py) `collect_and_merge()`가 신규 문서를 기존 목록 **끝에 append** → 305건 중 148~305번 위치에 저장.
2. [staged_report.py:148](../src/generators/staged_report.py) `_prepare_docs()`가 `documents[:MAX_DOCS]`(40건)로 **앞에서 절단** → 신규 문서는 전부 절단선 밖.
3. 그 결과 [staged_report.py:413](../src/generators/staged_report.py) `_added_doc_indices()`가 빈 리스트를 반환 → `stage2_patch`는 **기존과 똑같은 자료**에 "개정 지침" 메모만 붙여 챕터를 재작성.

**영향**: 검증 루프의 존재 이유(데이터 공백 보강)가 무효. agy Pro/Flash 호출, 갭 쿼리 재수집(10개 소스 × 쿼리 수), 신규 원문 fetch, Opus/Sonnet 챕터 재생성 비용이 전부 소모되지만 보고서의 데이터 공백은 그대로 남는다. "재작성했는데 여전히 '데이터 없음'"이 반복되는 구조.

**해결방안**:
- `_prepare_docs()`에서 `last_added_urls`에 기록된 문서를 `MAX_DOCS` 절단과 무관하게 **항상 포함**한다 (절단 후 누락분을 뒤에서 찾아 추가).
- 단, 현재 원문 매핑이 "문서 위치 번호 = manifest index"라는 암묵 가정에 묶여 있어(아래 M1), 이 수정과 함께 `_build_fulltext_index`를 **URL 키 매핑**으로 바꾸는 것이 안전하다. manifest에 URL이 이미 저장돼 있어 변경 비용이 작다.

### C2. "신뢰도 상위 40건" 선별이 보고서 입력을 메타데이터 스텁으로 도배한다

**현상**: 수소 런의 보고서 입력 40건 중 37건이 500자 미만(중앙값 202자). 특허 초록, R&D 과제명·기관·기간 목록, EIA 데이터셋 설명이 전부다. 논문·심층 웹 자료는 0건.

**원인**: 두 설계가 결합된 결과다.
1. [trust.py](../src/processors/trust.py)의 등급이 **출처의 권위만 보고 내용의 충실도는 보지 않는다**. `kipris.or.kr`(특허) S급, `odcloud.kr`(가스안전 R&D 목록) S[94], `eia.gov`(라우트 스텁) S급 — 모두 수백 자짜리 메타데이터인데, 수천~수만 자 원문이 확보된 A급 논문(75점)보다 항상 위에 정렬된다.
2. `_prepare_docs()`의 `[:MAX_DOCS]` 절단이 이 정렬을 그대로 받아들인다. 소스가 4개이던 시절(MAX_DOCS=40 설계 당시)에는 문제가 없었지만, 소스가 10개로 늘고 정부 API가 쿼리당 S급을 수십 건씩 쏟아내면서 절단선 위가 메타데이터로 포화됐다.

**영향**: 보고서가 사실상 **라이브러리 PDF 2~3건 + 메타데이터 스텁**만으로 작성된다. 5각도 쿼리 전략으로 애써 확보한 영어 학술자료(CLAUDE.md의 핵심 목적)가 보고서 단계에서 전량 버려진다. 검증 단계에서 "데이터 공백" 지적이 많이 나오는 근본 원인 중 하나로 추정된다 — 자료가 없어서가 아니라 **있는데 안 쓰여서**.

**해결방안** (조합 권장):
- **소스 유형별 쿼터**: 보고서 입력에서 `stats`/`patent`/`report`(목록형 정부 API) 합계 상한을 두고(예: 특허 5 + 통계 3 + R&D 목록 5), 남은 슬롯은 원문 확보(`manifest status=ok`) 문서를 등급순으로 채운다.
- **선별 점수 보정**: 정렬 키를 `trust_score` 단독에서 `trust_score + 내용 충실도(원문 확보 여부, 텍스트 길이 구간)`로 보정. 등급 체계 자체는 그대로 두고 "보고서 투입 순위"만 별도 계산하면 기존 테스트에 영향이 없다.
- **outline에 더 많은 후보 제공**: `_doc_listing`은 문서당 ~190자라 120건을 줘도 outline 호출 1회 ~8K 토큰 수준. 후보를 넓혀 주고 챕터 배분(stage1)에서 자연 선별시키는 방법도 토큰 부담이 작다.

## 높은 우선순위 문제

### H1. gas_safety가 범용어 매칭으로 무관한 주제에 침투한다

**페로브스카이트 태양전지** 보고서의 top40에 **수소 R&D 과제 9건**이 S[94]로 들어가 있다.
원인: [gov_kr.py:80](../src/collectors/gov_kr.py)의 로컬 키워드 매칭이 쿼리를 어절로 쪼개 `"기술"`, `"동향"` 같은
범용어 하나만 겹쳐도 통과시키고, [relevance.py:129](../src/processors/relevance.py)가 정부 소스에 강신호
검사를 면제(매칭 ≥1)해 주기 때문. 두 관대함이 겹쳐 노이즈가 S급으로 직행한다.

**해결**: `gas_safety_search`의 매칭 어절에서 relevance.py의 `GENERIC_TERMS`를 제외하고 강신호 어절만 사용한다(기존 `strong_terms()` 재사용 가능). 강신호 매칭 0건이면 반환하지 않는다.

### H2. EIA 어댑터는 사실상 노이즈 생성기다

- `"수소", "연료전지"` 쿼리가 **petroleum(석유) 라우트로 매핑**되어 있다 ([eia.py:25](../src/collectors/eia.py)). 수소 보고서에 석유 개요 페이지가 S급으로 투입된다.
- 반환 내용이 "데이터셋 설명 + 날짜 범위 + 하위 경로 5개"뿐인 스텁인데, `published`가 최신이라 SS(95+)까지 받아 top40 상위를 점유한다 (수소 런 6건).

**해결**: 수소→petroleum 매핑 제거. EIA 스텁 문서는 등급을 B 수준으로 별도 처리하거나, 실제 데이터 조회(하위 라우트의 통계값)까지 구현하기 전에는 보고서 투입 후보에서 제외(참고자료 목록에만 유지).

### H3. 원문 fetch가 "쓰일지 안 쓰일지 모르는" 300건 전부를 다운로드한다

수집 전량(261~323건)에 대해 httpx 1차 + 실패분 crawl4ai 헤드리스 브라우저 2차 + wayback 3차를 수행한다.
런당 113~216MB, crawl4ai만 30~56건. 그러나 보고서가 원문을 읽는 것은 top40(그나마 대부분 스텁)뿐.
**fetch 시간·대역폭·봇차단 리스크의 ~87%가 사용되지 않는 자료에 지출**된다.

**해결**: fetch 대상을 "보고서 투입 후보(선별된 N건) + 검증 루프 신규 문서"로 한정한다. C2의 선별 로직을 fetch **앞**으로 옮기면 자연스럽게 해결된다 (수집 → 선별 → 선별분만 fetch → 생성). 폐기 후보 중 나중에 필요해지는 자료는 검증 루프의 incremental fetch가 이미 처리할 수 있는 구조다.

### H4. 쿼리 언어와 무관하게 10개 소스를 전부 호출한다

`collect()`는 쿼리마다 10개 소스를 모두 부른다. 9쿼리 런이면 90회 호출인데:
- 한국어 쿼리 → arXiv·Semantic Scholar·Exa·EIA는 거의 0건 (CLAUDE.md에 이미 문서화된 사실)
- 영어 쿼리 → KOSIS·가스안전공사·국회도서관은 거의 0건

**해결**: 쿼리에 한글 포함 여부로 소스를 라우팅한다(한국어 → Tavily·KOSIS·가스안전·KIPRIS국내·국회도서관 / 영어 → Tavily·Exa·arXiv·S2·EIA·KIPRIS해외). API 호출 약 40% 절감, 무료 쿼터(Tavily·Exa 월 1,000회) 보호, 수집 시간 단축.

### H5. 정부 API 문서가 쿼리 수만큼 중복 누적된다

가스안전공사는 쿼리당 최대 5건씩 반환하므로 9쿼리면 같은 데이터셋에서 최대 45건(실측 22~35건)이
들어온다. 모두 S[94]라 top40을 잠식한다(C2를 악화). KIPRIS 해외특허도 동일 패턴(실측 7~32건).

**해결**: 정부 목록형 소스는 쿼리 단위가 아니라 **런 단위 상한**을 둔다 (예: gas_safety 총 8건, kipris 합계 10건). `collect()`에서 소스별 누계를 세면 간단하다.

## 중간 우선순위 문제

### M1. 원문 매핑이 "문서 위치 = manifest index"라는 깨지기 쉬운 가정에 묶여 있다

`_doc_block(d, i+1, fulltext_index)`은 payload 내 위치로 manifest index를 조회한다. incremental fetch에서
이전에 `failed`였던 URL이 재시도되면 **새 index로 다시 등록**되어 위치↔index 정렬이 어긋날 수 있다
(현재는 MAX_DOCS 절단이 어긋난 구간을 가려서 증상이 안 보일 뿐). 문서 재정렬·재배치도 전부 금지되는
구조라 C1·C2 수정의 발목을 잡는다. → **URL 키 매핑으로 전환** (C1 해결과 묶어서 1회 작업 권장).

### M2. stage3가 참고자료 목록 때문에 라이브러리 전체를 다시 로드한다

[staged_report.py:564](../src/generators/staged_report.py) `stage3_assemble()`이 참고문헌 생성용으로
`_prepare_docs()`를 재호출 → 33MB·46MB PDF의 페이지 캐시(JSON 수십 MB)를 다시 읽고 페이지 선발까지
수행한다. 참고문헌에 필요한 것은 제목·URL·등급뿐이다. → raw JSON 메타데이터만 읽는 경량 경로로 분리.

### M3. Exa에 전문(full text)을 요청해 놓고 1,000자만 쓴다

[search.py:50](../src/collectors/search.py) `contents: {"text": True}`로 전문을 받아 `[:1000]`으로 버린다.
원문은 어차피 fetch 단계에서 다시 받는다. → `{"text": {"max_characters": 1000}}`로 요청 자체를 줄이면
Exa 크레딧·전송량 절약.

### M4. 동시 실행 가드 미구현 (기지 사항)

`output/locks/{slug}.lock` pidfile — project-context.md에 이미 기록된 잔여 과제. 검증 루프가 길어
(agy 최대 20분 × 2 + 챕터 7회 생성) 중복 실행 사고 확률이 낮지 않다.

### M5. pipeline.py의 `TRUSTED_DOMAINS`는 사실상 죽은 설정

`trusted_only=True`일 때만 쓰이는데 호출 경로가 없고, trust.py의 S/A_DOMAINS와 내용이 중복·불일치한다.
혼동 방지를 위해 제거하거나 trust.py로 통합 권장.
